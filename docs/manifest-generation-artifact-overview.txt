Title: Manifest Generation and Artifacts – Plain‑English Guide

Overview
- The “manifest” is a single JSON file that lists all Guides in this repo. It is generated by a Python script located at:
  scripts/generate_manifest/manifest_generator.py
- The script scans site/sfguides/src/ and extracts metadata (title, categories, summary, last updated) from each guide, then writes:
  site/sfguides/src/_shared_assets/quickstart-manifest.json

What is CI (Continuous Integration)?
- CI is a set of automated jobs that run when code changes happen (e.g., pull requests, merges).
- In our setup, CI builds the manifest during PRs/pushes (as an artifact for review) and publishes it after merges to master (to a dedicated “manifests” branch).

When the jobs run
- On PRs or pushes: CI runs the generator and uploads the JSON as a build artifact attached to that run (not committed to Git).
- On merges to master: CI runs the generator and commits the JSON to a dedicated branch named “manifests” (not to master).
- No automatic time‑based schedule by default. We can optionally add a weekly schedule (example below).

Where teammates get the manifest (two options)
1) From a PR run (per‑commit artifact)
   - In GitHub → Actions → choose the PR’s run → download the artifact named “quickstart-manifest-<commit-sha>”.
   - CLI example (requires repo access):
     gh run list --branch <your-branch> -L 1
     gh run download <run-id> -n quickstart-manifest-<commit-sha> -D ./artifact

2) From the manifests branch (stable URLs, after merges to master)
   - Latest (rolling): https://raw.githubusercontent.com/Snowflake-Labs/sfquickstarts/manifests/quickstart-manifest.json
   - Versioned (immutable): https://raw.githubusercontent.com/Snowflake-Labs/sfquickstarts/manifests/quickstart-manifest-<commit-sha>.json
   - Browse the branch: https://github.com/Snowflake-Labs/sfquickstarts/tree/manifests

Why master stays clean
- The artifact workflow only uploads files to the Actions run; it does not commit to Git.
- The publish workflow commits exclusively to the “manifests” branch using a Git worktree and a guard that prevents writing to master.
- .gitignore includes site/sfguides/src/_shared_assets/quickstart-manifest.json to avoid accidental local commits to master.

Optional weekly schedule (if desired)
- Add this to the publish workflow to regenerate weekly even without code changes (example: every Friday at 09:00 UTC):

  on:
    push:
      branches: [ master ]
    schedule:
      - cron: "0 9 * * 5"

Exactly which workflows are added
- .github/workflows/manifest-artifact.yml
  - Triggers: pull_request and/or push (typically to master/main)
  - Actions:
    • Checkout repo (full history)
    • Set up Python
    • Run: python3 "scripts/generate_manifest/manifest_generator.py"
    • Upload: quickstart-manifest.json (artifact named quickstart-manifest-<commit-sha>)
  - Permissions: contents: read (no Git writes)

- .github/workflows/manifest-publish-branch.yml
  - Trigger: push to master (and optional manual dispatch)
  - Actions:
    • Checkout repo (full history)
    • Set up Python
    • Run: python3 "scripts/generate_manifest/manifest_generator.py"
    • Use git worktree to commit ONLY to branch “manifests”:
        quickstart-manifest.json            (latest/rolling)
        quickstart-manifest-<commit-sha>.json (versioned/immutable)
  - Permissions: contents: write (writes only to “manifests”, not master)

How to run locally (sanity check)
1) From repo root:
   python3 "scripts/generate_manifest/manifest_generator.py"
2) Verify output exists:
   ls -lh site/sfguides/src/_shared_assets/quickstart-manifest.json
3) (Optional) Compare with the artifact downloaded from the PR run for the same commit.

FAQ
- Q: Is the published JSON the same content the script produces locally?
  A: Yes. CI runs the exact same script and writes the same quickstart-manifest.json content.

- Q: Can we avoid committing the JSON to master?
  A: Yes. We never commit it to master; we publish only to the “manifests” branch and to PR artifacts.

- Q: Can we schedule weekly refreshes without a merge?
  A: Yes. Add a cron schedule to the publish workflow (example above); it will regenerate and push to “manifests” on the schedule.
